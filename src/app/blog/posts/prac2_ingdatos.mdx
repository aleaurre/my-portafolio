### Resumen ejecutivo

<Text marginBottom="8">
  Esta práctica aborda un aspecto clave del ciclo de vida del software: lograr que un proyecto pueda
  ser accesible públicamente sin intervención manual en cada actualización.
</Text>

<Text>
  Para ello se aprovechan herramientas de automatización como GitHub Actions, junto con la
  simplicidad y eficiencia del framework <strong>MkDocs</strong>, logrando documentar y publicar
  un portafolio técnico directamente en <strong>GitHub Pages</strong>.  
  Cada nuevo cambio realizado en la rama <InlineCode>main</InlineCode> dispara un proceso de build
  y despliegue completamente automático, reforzando buenas prácticas de integración continua.
</Text>

---

## Objetivos y alcances

<Text>
  El propósito principal consistió en habilitar una versión online del portafolio académico con
  contenido mínimo pero funcional, utilizando un proceso de publicación confiable, reproducible
  y sin pasos manuales adicionales.  
  Se creó un repositorio público a partir del template asignado, se configuró GitHub Pages con
  <strong>Source: GitHub Actions</strong>, se completó la estructura inicial con About, navegación
  y primeras prácticas, y finalmente se verificó el pipeline de deploy junto a la correcta
  accesibilidad de la URL pública entregada como resultado final.
</Text>

## Desarrollo técnico

<Text>
  Se utilizó la opción <InlineCode>Use this template</InlineCode> en GitHub para crear un repositorio
  inicial con la estructura de MkDocs ya configurada. Se habilitó el despliegue mediante
  <strong>Settings → Pages → Source: GitHub Actions</strong>, estableciendo un pipeline totalmente
  automatizado.  
  Además se agregaron enlaces internos para asegurar navegación futura y se constató que cada
  actualización en la rama principal se publicara automáticamente sin participación humana,
  cumpliendo así con los principios de CI/CD desde el principio del proyecto.
</Text>

---

## Decisiones y fundamentos

<Text>
  Se mantuvo la estructura base del template para preservar orden y simplicidad en la gestión del
  contenido estático. Se evitó el uso de la rama <InlineCode>gh-pages</InlineCode> en favor de una
  automatización más robusta mediante GitHub Actions.  
  Los enlaces relativos fueron seleccionados para prevenir errores derivados de posibles cambios de
  nombre en el repositorio o en la ruta del sitio generado.  
  El enfoque se centró en una integración mínima viable que permitiera escalar el portafolio
  rápidamente, manteniendo tiempos de entrega breves y asegurando una experiencia profesional desde
  el inicio.
</Text>

## Reproducibilidad local

<Text marginBottom="12">
  El entorno necesario es sencillo: basta con contar con Python, MkDocs y conexión a GitHub para
  continuar iterando en el sitio localmente.
</Text>

```bash
pip install -q mkdocs mkdocs-material
mkdocs serve   # http://127.0.0.1:8000
```

<Text marginTop="12"> No es necesario ejecutar comandos de publicación: GitHub Actions se encarga automáticamente tras cada commit en la rama principal. </Text>
