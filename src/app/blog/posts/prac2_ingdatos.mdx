### Resumen ejecutivo

<Text marginBottom="8">
  Esta práctica aborda un aspecto clave del ciclo de vida del software: lograr que un proyecto
  pueda ser accesible públicamente sin intervención manual en cada actualización.
</Text>

<Text>
  Para ello se aprovechan herramientas de automatización como GitHub Actions, junto con la
  simplicidad y eficiencia del framework <strong>MkDocs</strong> para documentar y publicar un
  portafolio técnico hospedado directamente en <strong>GitHub Pages</strong>.  
  Cada nuevo cambio realizado en la rama <InlineCode>main</InlineCode> dispara un proceso de build
  y despliegue completamente automático, reforzando buenas prácticas de integración continua.
</Text>

---


## Objetivos y alcances

<Text marginBottom="16">
  El propósito principal fue habilitar una versión online del portafolio académico, con información
  mínima pero funcional, asegurando un proceso de publicación confiable, reproducible y sin pasos
  manuales adicionales.
</Text>

<List>
  <ListItem>Crear un repositorio público desde el template asignado</ListItem>
  <ListItem>Configurar GitHub Pages con <strong>Source: GitHub Actions</strong></ListItem>
  <ListItem>Completar el contenido inicial: About, navegación y primeras prácticas</ListItem>
  <ListItem>Verificar el pipeline de deploy y correcto funcionamiento del sitio</ListItem>
  <ListItem>Publicar la URL final para su entrega</ListItem>
</List>

## Planificación y ejecución

| Actividad | Estimado | Real | Nota |
|---|:---:|:---:|---|
| Crear repo desde template | 10m | 5m | Configuración básica |
| Activar GitHub Pages + Actions | 5m | 5m | Deploy automatizado |
| Contenido mínimo inicial | 45m | 20m | About + navegación |
| Commit / Push y verificación | 10m | 5m | Pipeline verde |
| Entrega de URL pública | 5m | 10m | Validación externa |

<Text marginTop="12">
  <strong>Resultado:</strong> publicación garantizada en menos de una hora, con flujo automatizado listo para iterar.
</Text>

## Desarrollo técnico

<Text marginBottom="12">
  El proceso se centró en establecer un canal fluido desde la edición local hasta la visibilidad
  pública en la web:
</Text>

<List>
  <ListItem>Se utilizó la opción <InlineCode>Use this template</InlineCode> en GitHub para crear el repositorio inicial</ListItem>
  <ListItem>Se habilitó la página en <strong>Settings → Pages → Source: GitHub Actions</strong></ListItem>
  <ListItem>El template de MkDocs brindó estructura de navegación y estilo uniforme</ListItem>
  <ListItem>Se agregaron enlaces internos para futuras visualizaciones y prácticas técnicas</ListItem>
  <ListItem>Cada actualización en <InlineCode>main</InlineCode> publica automáticamente el sitio</ListItem>
</List>

## Decisiones y fundamentos

<Text marginBottom="12">
  Varias elecciones de diseño y configuración aseguraron la escalabilidad del proyecto:
</Text>

<List>
  <ListItem>Mantener la estructura original del template: <InlineCode>docs/</InlineCode> + <InlineCode>assets/</InlineCode></ListItem>
  <ListItem>Evitar despliegues manuales usando la rama <InlineCode>gh-pages</InlineCode></ListItem>
  <ListItem>Uso de enlaces relativos para prevenir errores en URLs</ListItem>
  <ListItem>Despliegue automático ante cada push → enfoque CI/CD</ListItem>
  <ListItem>Integración mínima viable para favorecer iteración rápida</ListItem>
</List>

## Reproducibilidad local

<Text marginBottom="12">
  El entorno necesario es sencillo: Python, MkDocs y conexión a GitHub.
</Text>

```bash
pip install -q mkdocs mkdocs-material
mkdocs serve   # Servidor local: http://127.0.0.1:8000
```
<Text marginTop="12"> No es necesario ejecutar comandos de publicación: GitHub Actions se encarga automáticamente tras cada commit. </Text>

## Reflexión personal
<Text> La principal enseñanza de esta práctica es la importancia de automatizar los procesos repetitivos. 
El simple hecho de que cada actualización se publique sin pasos manuales minimiza errores, reduce la fricción al iterar 
y habilita una mentalidad más profesional alineada con flujos reales de DevOps. </Text>
